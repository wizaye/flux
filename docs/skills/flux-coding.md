# Skill: Flux Coding (default)

> Activate this skill for any code-writing task in this repo.

## When to use

Default skill. Always loaded.

## What to do

1. **Read first, code second.**
   - Find the feature in `docs/feature_spec.md`.
   - Find the matching sequence diagram in `docs/architecture.md` §3.
   - Confirm dependencies in `docs/tech_stack.md`.
   - Confirm no rule violations in `docs/LLM_SYSTEM_INSTRUCTIONS.md` §1 and
     `docs/LLM_ANTIHALLUCINATION.md`.

2. **Plan in 3–5 bullets** before writing code. Mention:
   - The files you'll touch
   - The Tauri command(s) you'll add or change
   - The DB tables or migrations involved
   - The UI surface affected
   - The test you'll add

3. **Code.** Follow the conventions:

   **Rust**
   - One crate per bounded context (`zenvault-domain`, `zenvault-db`,
     `zenvault-fs`, `zenvault-ai`, `zenvault-plugins`, app crate `src-tauri`).
   - Public async functions get `#[tracing::instrument(skip(big_args), err)]`.
   - Errors: per-crate `thiserror` enum, mapped to top-level `AppError`.
   - DB access: borrow from the pool, run on `spawn_blocking`. Never hold
     the connection across `.await`.
   - Tests: `#[cfg(test)] mod tests { ... }` next to code; integration tests
     in `tests/` use a temp vault.

   **TypeScript**
   - Use `bindings.ts` generated by tauri-specta — never hand-write a DTO.
   - State: Zustand for UI state, React Query for vault data.
   - Editor: raw CM6 (`@codemirror/*`) inside `useEffect`. Destroy on unmount.
   - Components: functional, hooks-only, Radix primitives + shadcn/ui patterns.
   - Lists > 100 rows: virtualize with `@tanstack/react-virtual`.

4. **Verify.** Run the relevant verified command from
   `docs/LLM_SYSTEM_INSTRUCTIONS.md` §4. Don't claim "done" without it.

5. **Document.** If behavior changed, update `docs/feature_spec.md`. If a
   dependency changed, add `docs/adr/NNN-<title>.md`.

## Output format

- **Diffs / patches**, not full files when editing.
- **File-line citations** when referring to existing code.
- **Concise prose**; no preambles.

## Stop conditions

Stop and ask the user when:

- The change touches more than one feature area in `feature_spec.md`.
- A required dependency isn't in `tech_stack.md`.
- A rule in `LLM_SYSTEM_INSTRUCTIONS.md` §1 would have to bend.
- You can't verify a library/API signature.
